## TL;DR

Three threads bundled into v0.0.5:

1. **Artifact output contract.** Every visual-output skill produces both `.html` (browser preview) and `.pdf` (print/email-ready) by default. The headless-Chrome `render-pdf.sh` script (shipped in v0.0.4 for /listing-presentation only) gets promoted to a shared `scripts/render-pdf.sh`. `/cma` seller and buyer both invoke it. CLAUDE.md documents the contract so future visual skills inherit it automatically.

2. **State scan contract.** Every intake/onboarding skill scans the user's existing realty-stack state (`~/.config/realty-stack/` + relevant locations) BEFORE asking the user for material. Pulls relevant content from sibling artifacts where it overlaps. Realtor only fills gaps. The more they accumulate, the easier the next setup gets. CLAUDE.md documents the contract so future intake skills inherit it.

3. **Buyer-presentation skill pair.** `/buyer-presentation-template` (one-time consultative builder, mirrors listing-pres-template with buyer-side framing) + `/buyer-presentation` (per-appointment HTML+PDF pitch). Both implement the new state scan contract — pulling shared sections from `/listing-presentation-template` if it exists.

## Motivation

**Artifact contract.** The v0.0.4 listing-presentation e2e exposed that "Cmd+P → Save as PDF in browser" is not viable for non-technical realtors. The headless-Chrome script solved it for /listing-presentation. Shipping /cma without the same treatment creates a confusing asymmetry — realtor runs /cma after /listing-presentation, gets HTML only, has to know what Cmd+P does. Canonicalize the contract.

**State scan contract.** When a realtor sets up `/listing-presentation-template`, then later runs `/buyer-presentation-template`, they're going to repeat themselves on sections that have nothing to do with seller-vs-buyer (About Me, Why Hire Me, Track Record, Testimonials, Pricing approach, Fee structure). Re-typing is anti-product. The compounding value of accumulated state is the whole point of "Realty Stack" as a system — every artifact created should make the next setup easier.

**Buyer-presentation pair.** Originally deferred from v0.0.4 to "validate the listing-side pattern first." Pattern is validated. Build the buyer-side now, both because realtors need it AND because it's the first real test of the state scan contract (90% of buyer-pres-template content comes from listing-pres-template).

## Sub-project A — Artifact output contract

### Design

**Canonical contract (new section in CLAUDE.md):**

> Every Realty Stack visual-output skill (any skill that produces a presentation, listing flyer, social card, property website, or other shareable visual artifact) MUST produce BOTH:
> - `.html` — browser preview, interactive, full @media screen layout
> - `.pdf` — print/email-ready, @media print layout, generated via `scripts/render-pdf.sh` (headless Chrome)
>
> Both files land in `~/Downloads/` with the same slug, different extensions. The skill confirms BOTH paths back to the realtor in a single message.
>
> **Graceful fallback when Chrome is missing:** the render-pdf.sh script exits with an error. The skill catches it, writes HTML anyway, tells the realtor: *"Saved HTML to ~/Downloads/{slug}.html. Couldn't auto-generate PDF — install Chrome from google.com/chrome to get auto-PDF, or open the HTML in any browser and Cmd+P → Save as PDF."*
>
> **Why both:** HTML is for browser preview (interactive widgets in /cma net sheet / mortgage calc; tab navigation). PDF is for email attachment and printing (seller leaves with a copy after the appointment). Realtors don't know what Cmd+P does — auto-PDF removes that friction entirely.

## Sub-project B — State scan contract

### Design

**Canonical contract (new section in CLAUDE.md):**

> Every Realty Stack intake/onboarding skill (any skill that asks the realtor for material — content, settings, preferences, brand details, voice samples) MUST scan the user's existing Realty Stack state BEFORE asking for material.
>
> **Canonical scan locations:**
> - `~/.config/realty-stack/voice-profile.md` — agent name, brokerage, primary market, signoffs, voice samples
> - `~/.config/realty-stack/brand-kit.md` — wordmark, colors, fonts, tagline, asset paths
> - `~/.config/realty-stack/listing-presentation-template.md` — listing pitch content
> - `~/.config/realty-stack/buyer-presentation-template.md` — buyer pitch content
> - `~/.config/realty-stack/brand-assets/` — base64-embedable visual assets (logo, headshot)
> - `~/Downloads/*-listing-presentation.{html,pdf}` — past listing presentations (agent could mine for stats, sample addresses)
> - `~/Downloads/*-cma.{html,pdf}` — past CMAs (agent's recent comp work)
>
> **The scan-and-inherit workflow** (Step 0 of every intake skill):
> 1. Scan canonical locations. Build an inventory of existing artifacts.
> 2. Identify what's relevant to THIS skill's intake.
> 3. Report findings to the realtor in plain English. Example: *"Found your voice profile, brand kit, and listing-presentation-template. I'll pull About Me, Why Hire Me, Track Record, Testimonials, Pricing, and Fees from there — that covers 6 of the 8 standard sections. Want me to ask about the 2 buyer-specific sections (How I Find Homes, Negotiation Philosophy), or do you have material to drop in?"*
> 4. If realtor confirms inheritance, pre-populate the section structure with the inherited content. The realtor can edit during the refinement loop (Step G).
> 5. For sections NOT covered by inheritance, fall through to the existing intake workflow.
>
> **Why:** the system gets smarter as it accumulates. Realtors should never re-type information that already exists in their realty-stack state. Compounding value of state is the whole product thesis.

- Modify: `CLAUDE.md` — add "State scan contract" section
- Modify: `skills/listing-presentation-template/SKILL.md` — add Step 0 (state scan); reorganize Step A to be a fallback when nothing inheritable was found
- Modify: `skills/brand-kit-capture/SKILL.md` — add Step 0 (state scan — pull agent name/brokerage from voice profile if exists)
- Modify: `skills/voice-draft/SKILL.md` — add Step 0 (state scan — pull agent name/brokerage from brand kit if exists)

### Scan workflow detail (canonical Step 0)

```
Step 0 — State scan

Before asking the realtor for material, scan canonical locations:

  ~/.config/realty-stack/voice-profile.md
  ~/.config/realty-stack/brand-kit.md
  ~/.config/realty-stack/listing-presentation-template.md
  ~/.config/realty-stack/buyer-presentation-template.md
  ~/Downloads/*-listing-presentation.{html,pdf}
  ~/Downloads/*-cma.{html,pdf}

For each found artifact, identify what's relevant to THIS skill's intake.
Build a "found / relevant / inheritable" inventory.

Report to realtor:

  "Found: {list of artifacts found}
  Relevant to this skill: {list with one-line per item explaining the pull}

  I can pre-populate {X of N} sections / fields from these. Want me to do
  that, then ask about the remaining {N-X}?"

If yes → pre-populate, jump to Step D (gap-fill) for the remaining items.
If no → fall through to the full intake (Step A — front-loaded data dump).
```

## Sub-project C — Buyer-presentation skill pair

### Design

Mirrors `/listing-presentation-template` + `/listing-presentation` (the v0.0.4 listing-side pair). Buyer-side framing applied to sections.

### `/buyer-presentation-template`

Same Steps A–H workflow as listing-presentation-template + new Step 0 scan.

**8 standard sections (buyer-side):**
1. About Me — same as listing-side (inheritable)
2. Why Hire Me — same as listing-side, with buyer-side framing where relevant (inheritable, with light edit pass)
3. My Buying Process — phases from initial consult through closing (buyer-specific)
4. How I Find Homes — search philosophy, off-market access, alerts (buyer-specific)
5. Recent Track Record — recent buyer-side closings (could inherit listing-side track record if agent works both sides + edit)
6. Testimonials — buyer testimonials preferred; can inherit listing-side as "Verified Client" if needed
7. Negotiation Philosophy — buyer-side framing (vs Pricing Philosophy on listing side)
8. Buyer Representation Agreement — fee structure for buyer side (vs Fee Structure on listing side)

**4-tab HTML output:**
- Tab 1: Who I Am (About Me, Why Hire Me)
- Tab 2: How I Work (My Buying Process, How I Find Homes)
- Tab 3: Track Record (Recent Track Record, Testimonials)
- Tab 4: Working Together (Negotiation Philosophy, Buyer Rep Agreement)

**Template file format:** mirror listing-presentation-template.md format, stored at `~/.config/realty-stack/buyer-presentation-template.md`.

### `/buyer-presentation`

Same Steps 1–5 workflow as listing-presentation.

**Per-listing personalization:**
- Buyer name(s) — "Prepared for [Buyer Name]"
- Target property address — "Touring [Address]" or generic "Your Home Search" if no specific property
- Appointment date
- Optional custom note

**Optional inline /cma offer at Step 3** — for buyer-side CMA (offer strategy analysis).

**Output:** HTML + PDF per the new artifact contract.

## Open questions for implementation

None blocking design. Implementation-detail decisions:

1. **`render-pdf.sh` SKILL invocation path.** Once moved to `scripts/` (repo root), the canonical reference is `${CLAUDE_PLUGIN_ROOT}/scripts/render-pdf.sh`. Confirm during implementation.

2. **State scan implementation in each skill.** Bash `ls`/`find` vs structured tool — bash is fine. Just enumerate the canonical locations + check existence + report.

3. **What sections are "inheritable" between listing-pres-template and buyer-pres-template.** Lean: About Me (100%), Why Hire Me (90%, light edit), Recent Track Record (75% — buyer-side closings may differ), Testimonials (90% — agent decides per-testimonial). Marketing Philosophy / Pricing / Fee Structure → NOT inheritable (buyer-side has different equivalents).

4. **`/cma` buyer path PDF rendering.** The mortgage calc is interactive — at PDF render time, it shows whatever default values were set. That's the desired behavior (frozen snapshot for the appointment). Confirm with Holden during e2e.

5. **Step 0 scan UX when inventory is empty.** Skip the scan output entirely; fall through silently to Step A. Don't surface "I scanned and found nothing" — feels noisy.

## Success criteria

1. `/cma` produces both HTML + PDF on next invocation (same as `/listing-presentation` does today).
2. Realtor running `/buyer-presentation-template` after `/listing-presentation-template` sees a "Found your listing-presentation-template — I'll pull these 6 sections" prompt, accepts, and only has to provide buyer-specific content for the remaining 2 sections.
3. Realtor running `/voice-draft` with a brand kit already in place doesn't get re-asked for agent name/brokerage.
4. Realtor running `/brand-kit-capture` with a voice profile already in place doesn't get re-asked for agent name/brokerage.
5. All 8 skills (4 existing intake skills + 4 visual-output skills) follow the appropriate canonical contract (state scan / artifact output) per their type.
6. CLAUDE.md is the canonical source for both contracts; future skills inherit them automatically.
